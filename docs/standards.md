# 🛠️ 软件版本控制与管理规范 (SemVer)

在 AI 项目开发中，版本号不仅仅是数字，它记录了 **模型迭代、数据扩容与功能演进** 的生命周期。本项目遵循语义化版本规范。

---

## 1. 语义化版本格式：X.Y.Z

| 组成部分 | 名称 | 触发条件 | 例子 |
| :--- | :--- | :--- | :--- |
| **X** (Major) | **主版本号** | **不兼容**的重大改动或架构重构 | 从纯向量库改为向量+知识图谱混合架构 |
| **Y** (Minor) | **次版本号** | **向下兼容**的新功能或数据扩容 | 接入宋词语料；增加“典故探测”功能 |
| **Z** (Patch) | **修订号** | **向下兼容**的问题修复或性能微调 | 修复搜索结果显示错位；优化加载速度 |

> **开发准则**：处于 `0.x.x` 阶段时，代表项目仍在初始开发期（MVP）。一旦发布 `1.0.0`，代表产品已具备稳定的功能闭环与演示精度。

---

## 2. 预发布版本标签 (Pre-release)

在版本号后添加连字符，标注当前开发的成熟度：

* **`alpha` (内测版)**：`v0.1.0-alpha`
    * 功能不完整，主要用于开发者自我验证核心链路。
* **`beta` (公测版)**：`v0.5.0-beta`
    * 功能冻结 (Feature Frozen)，主要用于寻找 Bad Case 和稳定性测试。
* **`rc` (发布候选)**：`v1.0.0-rc.1`
    * 除非发现致命 Bug，否则将作为正式版本发布。

## 4. 工程师的自我修养
1. **Git Tag**: 建议每次发布节点使用 `git tag` 标记版本。
2. **UI 展示**: 在界面侧边栏标注当前 Version，增加项目的专业感知度。
3. **数据挂钩**: 每一个 Minor (Y) 版本的更新，需注明对应的向量库规模（如：500k -> 1M）。
