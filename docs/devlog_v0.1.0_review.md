# 🏮 开发复盘：从典故 Agent 到语义搜索 MVP (v0.1.0)

## 1. 项目起源与定位演变 (Project Evolution)
* **初始设想**：构建一个“宋词典故 Agent”，利用 LLM 识别并解释词中的隐喻和历史典故。
* **现实瓶颈**：典故识别属于 NLP 中的深层任务，需要极高的实体链接（Entity Linking）精度。在缺乏结构化典故数据集的情况下，直接硬做 Agent 容易产生幻觉。
* **路径修正**：将目标转向**语义意象检索**。
    * **核心逻辑**：如果能实现“搜索意境，召回诗句”（如搜“红蜡”能召回“红蜡泪”），实际上就解决了典故匹配中最难的“模糊语义关联”问题。
* **最终方案**：确定了 **RAG (检索增强生成)** 的技术路线，先完成高性能检索（Retrieval）基建。

---

## 2. 语料库的“MVP 切换”策略 (Data Strategy)
在 v0.1.0 阶段，项目采取了“曲线救国”的数据策略：
* **最终目标**：处理 `chinese-poetry` 项目中的全量宋词（JSON 格式）。
* **MVP 方案**：先行接入另一 `poetry` 项目提供的**唐诗 CSV 语料**。
* **决策理由**：
    1.  **工程解耦**：`chinese-poetry` 的宋词数据为多文件、异构 JSON，清洗成本高。
    2.  **快速验证**：唐诗 CSV 格式规整（标题/作者/内容已对齐），适合作为“第一块敲门砖”来验证 `BERT-CCPoem` 向量模型的召回效果。
    3.  **迭代逻辑**：先在唐诗上跑通“向量提取-索引构建-UI展示”的全链路，再在 v0.2.0 中解决宋词的 ETL（抽取、转换、加载）难题。

---

## 3. 技术栈权衡：为什么选 FAISS + CSV？ (Tech Stack Decision)
在向量数据库选型上，本项目放弃了 ChromaDB 等重型方案，选择了 **CSV (元数据) + FAISS (向量索引)** 的轻量级组合：
* **透明性**：CSV 文件人类直观可读，方便在开发初期快速核对“检索结果”与“原始文本”的一致性。
* **低开销**：FAISS 属于本地二进制索引，无需启动额外的 Docker 容器或后台进程，极大地简化了本地开发环境。
* **性能匹配**：对于百万级以下的数据量，FAISS 的 `IndexFlatL2` 提供了毫秒级的响应速度，且更便于进行索引策略的底层调优。

## 4. 工程化与版本规范 (Engineering Standards)
* **代码与数据分离**：确立了“代码上云，数据落地”的原则。通过 `.gitignore` 严格控制仓库体积，确保 GitHub 仓库只承载核心逻辑。
* **规范化提交**：引入 **Conventional Commits** 规范（如 `feat:`, `fix:`, `docs:`），为后续多版本迭代（如 v0.2.0 宋词接入）建立清晰的追踪链路。
* **文档基建**：同步建立 `Backlog`（需求池）和 `DevLog`（复盘记录），将零散的灵感转化为结构化的工程计划。
